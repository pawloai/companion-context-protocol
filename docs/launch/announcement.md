# Announcement Copy

Status: Draft

Use this copy for a public feedback preview. Do not describe CCP as a finished standard, adopted protocol, or design-partner-reviewed artifact.

## One-Paragraph Positioning

Companion Context Protocol is a draft protocol proposal for permissioned companion-animal context exchange. It currently includes canonical JSON Schemas, synthetic examples, conformance tests, and adapter sketches for narrow commerce, care-facility, pickup-verification, and care-network lookup flows. The repository is ready to ask design partners for review, but it is not yet an adopted standard or endorsed interoperability baseline.

## Short Announcement

```text
I am opening a draft of Companion Context Protocol for feedback.

CCP is an open, transport-neutral protocol proposal for permissioned companion-animal context exchange. It is not a standard yet. The current draft is a set of schemas, examples, tests, and adapter sketches that need outside review before anyone should treat it as an interoperability baseline.

The first schema-backed profile is intentionally narrow: commerce-safe pet context for product recommendation and filtering, without exposing staff notes, wellness timelines, diagnosis history, billing data, or unrelated owner data. The draft also includes narrower Care Facility boarding-preparation, pickup-verification, and Care Network lookup slices.

The repo includes a draft spec, canonical JSON Schemas, examples, conformance tests, OpenAPI and MCP adapter sketches, and draft TypeScript/Python helper packages.

I am looking for design-partner feedback from pet retailers, care operators, facility software builders, practice-management software builders, vet-adjacent software teams, insurers, registries, agent tooling builders, and pet data platforms:

- Would you implement this Commerce Context Profile?
- Would you implement the first Care Facility boarding-preparation slice?
- Is the Facility Truth Profile a more urgent first review target for agent accuracy around hours, services, eligibility, certifications, and contact or booking methods?
- Are the scope and purpose boundaries clear?
- What context is missing or too sensitive?
- What prior art or existing ecosystem constraint is missing?
- Can you validate compatibility with the current tests?

This is a draft for review, not a stable standard or endorsed protocol.
```

## Feedback Request

```text
If you work on pet commerce, pet care operations, facility software, practice-management software, vet-adjacent software, insurance, registries, agent tooling, or pet data infrastructure, I would value focused feedback on this draft.

The most useful feedback is concrete:

- What would block implementation?
- What field is missing?
- What field feels unsafe?
- Which adapter would you use?
- Which existing standards, platforms, or operational workflows does the draft fail to account for?
- Should Facility Truth be reviewed before pet-specific profiles?
```
